# LightLocalTranslator — Полный план реализации

> **Ветка:** `light-local-translator`
> **Тяжёлые модели (>4B):** тестируются параллельно на отдельном ПК (не в этой ветке)

## Контекст и ограничения

**Цель:** офлайн-переводчик для слабых ПК (CPU-only, 8–16 GB RAM, без GPU) с обязательным сохранением форматирования документов.

**Стратегия:**
- Сначала тестируем **готовые решения**. Если одно побеждает по всем критериям → оно и есть финальный LightLocalTranslator, берём как есть, без пересборки.
- Только если ни одно готовое решение не устраивает → собираем своё.

**Правила:**
- Никакое дополнительное ПО (Ollama, LM Studio и аналоги) без явного согласования
- Git-клонирование репозиториев — ок
- Java — можно установить (нужна для Okapi)
- Скачивание моделей — ок, подключаем к существующему или своему коду
- Форматирование документов **обязательно** сохраняется для всех форматов: PDF (searchable), MD, DOCX, XLSX, TXT

**Разработка ведётся на:** Windows 10, Python 3.11, RTX 4080 (16 GB)
**Целевое железо:** CPU-only ПК, 8–16 GB RAM

## Фаза 1 — Тестирование готовых решений

Цель: понять потолок качества и удобства готовых пакетов **без написания своего кода**.

### 1.1 Argos Translate

**Установка:** `pip install argostranslate`
**Что это:** OpenNMT-based, офлайн, пакеты моделей скачиваются через API.
**Тестируем:**
- Установка пакета языковой пары ru↔en
- Перевод блоков A–C из тестового корпуса
- Замер: скорость (сек/абзац), RAM в пике, субъективное качество 1–5

**Ожидаемый результат:** простейший baseline, качество умеренное.

### 1.2 LibreTranslate (self-hosted)

**Установка:** `pip install libretranslate`
**Что это:** веб-сервер с REST API поверх Argos (тот же backend).
**Тестируем:**
- Запуск локального сервера (`libretranslate --host 0.0.0.0`)
- Перевод через API блоков A–C
- Удобство интеграции в собственный код

**Ожидаемый результат:** то же качество что Argos, но с удобным API.

### 1.3 Document-Translation + Okapi

**Установка:**
1. Установить Java (JDK 11+)
2. Скачать Okapi Framework (Rainbow / Tikal)
3. `git clone https://github.com/kukas/document-translation`
4. Подключить MT-бэкенд (OPUS-MT или NLLB через локальный API)

**Что это:** единственное готовое решение с **нативным** сохранением форматирования. Okapi Framework извлекает переводимые сегменты из DOCX/ODT/HTML/XML/RTF/PDF, передаёт их в MT-бэкенд, и вставляет обратно с полным сохранением разметки.

**Почему приоритет:** решает задачу форматирования документов лучше всех остальных готовых стеков. Это эталон, с которым будем сравнивать свой обработчик.

**Тестируем:**
- Полный round-trip: `test_doc.docx` → перевод → сохранённый DOCX
- Качество сохранения: таблицы, стили, колонтитулы, нумерация
- PDF: проверить что searchable PDF сохраняется как searchable
- Качество перевода (зависит от бэкенда — тестируем с OPUS-MT и NLLB-600M)
- Скорость на документе 3000 слов

**Ожидаемый результат:** лучшее сохранение форматирования из всех готовых решений. Становится эталоном для нашего собственного `docx_handler.py` и `pdf_handler.py`.

> **Вывод по фазе 1 — дерево решений:**
> ```
> Готовое решение побеждает по качеству + форматированию?
> ├── ДА → это и есть финальный LightLocalTranslator, берём как есть. СТОП.
> └── НЕТ → переходим в Фазу 2 (лёгкие модели напрямую)
> ```
> - Okapi тестируем **в любом случае** — он эталон по форматированию, даже если его качество перевода слабее.
> - Argos/LibreTranslate скорее всего не победят по качеству, но дают быстрый baseline.

## Фаза 2 — Тестирование лёгких моделей напрямую

> Сюда переходим только если ни одно готовое решение из Фазы 1 не устроило.

Цель: найти лучшее соотношение качество / скорость / RAM для CPU-only машины.
Все запускаются через Python без сторонних серверов.

### Приоритет тестирования

| # | Модель | Размер (диск) | RAM CPU (int8/Q4) | Runtime | Приоритет |
|---|--------|--------------|-------------------|---------|-----------|
| M1 | **NLLB-200-distilled-600M** | ~1.2 GB | ~1.5 GB | CTranslate2 | ★★★ первый |
| M2 | **OPUS-MT ru-en / en-ru** | ~300 MB | ~0.5 GB | CTranslate2 | ★★★ первый |
| M3 | **TranslateGemma 4B Q4_K_M** | ~2.5 GB GGUF | ~3–4 GB | llama-cpp-python | ★★★ первый |
| M4 | **MADLAD-400-3B-MT** | ~3 GB | ~3.5 GB (int8) | CTranslate2 int8 | ★★ второй |
| M5 | **M2M100-1.2B** | ~5 GB | ~5.5 GB | CTranslate2 | ★ третий |
| M6 | **TowerInstruct-Mistral-7B Q4** | ~4.5 GB GGUF | ~5–6 GB | llama-cpp-python | ★ третий |

### Что тестируем для каждой модели

Для каждой модели запускаем одинаковый benchmark:

```
блок A: 50 слов (EN→RU, RU→EN)
блок B: 30 предложений
блок C: 10 абзацев ~200 слов (технические + научные)
блок D: 1 документ ~3000 слов (plain text)
```

Замеряем:
- chrF2++ (через `sacrebleu`, где есть референс)
- Скорость: сек/абзац, сек на документ 3000 слов
- RAM: пиковое потребление (psutil)
- Субъективная оценка: 1–5 по 10 примерам

### Порядок запуска

```
M2 (OPUS-MT) → M1 (NLLB-600M) → M3 (TranslateGemma 4B) → M4 (MADLAD-3B) → M5 → M6
```

Начинаем с самых лёгких, чтобы быстро получить нижнюю и среднюю планку.

```

│   ├── backends/

│   │   └── llamacpp_backend.py      # TranslateGemma GGUF

├── formats/

│   ├── docx_handler.py    # DOCX (python-docx)
│   ├── pdf_handler.py     # PDF (PyMuPDF)

### 3.2 Обработка форматов — правила сохранения

#### TXT
- Разбивка на абзацы по `\n\n`, перевод каждого, склейка обратно.
- Пустые строки и отступы сохраняются.

#### Markdown
- Парсим через `markdown-it-py` в AST.
- Защищаем (не переводим):
  - блоки кода ` ``` ` и inline-код `` ` ``
  - URL и пути в ссылках `[текст](url)` — переводим только `текст`
  - HTML-теги
  - YAML frontmatter (если есть)
  - Синтаксис таблиц — переводим только содержимое ячеек
- Восстанавливаем исходные маркеры разметки вокруг переведённого текста.

#### DOCX
- Итерируем по `paragraph.runs` через `python-docx`.
- Каждый run переводится **отдельно** — сохраняются bold, italic, underline, font, size.
- Таблицы: итерация по `table.rows → row.cells → cell.paragraphs`.
- Колонтитулы, сноски — обходим аналогично.
- Стили, нумерация, поля страницы — не трогаем (сохраняются автоматически).

#### PDF (searchable)
Используем PyMuPDF (`fitz`):
1. Открываем PDF, получаем `page.get_text("blocks")` — список блоков с координатами.
2. Для каждого блока: переводим текст, сохраняем bbox (позицию).
3. Перезаписываем: закрашиваем оригинальный текст через `page.add_redact_annot(bbox)` → `page.apply_redactions()`.
4. Вставляем переведённый текст через `page.insert_textbox(bbox, translated, fontsize=...)`.
5. Сохраняем как searchable PDF (`doc.save()`).

> **Ограничение:** PDF со сканами (не searchable) — пропускаем на данном этапе (нужен OCR).

#### XLSX
- Итерируем через `openpyxl` по всем листам, строкам, ячейкам.
- Переводим только ячейки со строковым значением (`cell.data_type == 's'`).
- Формулы (`=SUM(...)`) — **не трогаем**.
- Числа, даты — **не трогаем**.
- Стили ячеек, цвета, шрифты, объединения — сохраняются (openpyxl хранит их отдельно от значений).
- Именованные диапазоны, фильтры, сводные таблицы — не трогаем.

### 3.3 Chunker (разбивка длинных текстов)

Для моделей с ограниченным контекстом (NLLB, OPUS-MT):
- Разбивка на сегменты по предложениям (не рубим на полуслове).
- Максимальный размер сегмента: 512 токенов для NMT, 2048 для GGUF.
- Склейка результатов с сохранением пробелов/переносов.

### 3.4 GUI (Gradio)

Два режима:
1. **Текст** — textarea ввода, выбор языков, кнопка перевести, textarea вывода.
2. **Файл** — drag & drop файла (TXT/MD/DOCX/PDF/XLSX), выбор языков, скачать результат.

Дополнительно:
- Выбор бэкенда (модели) в настройках
- Индикатор прогресса для длинных документов
- Лог с количеством сегментов и временем

## Метрики финального тестирования

| Метрика | Инструмент | Для чего |
|---------|-----------|---------|
| chrF2++ | `sacrebleu` | Качество перевода текста |
| Скорость (сек/документ) | `time.perf_counter()` | Практичность на CPU |
| RAM пик (MB) | `psutil.Process().memory_info()` | Пригодность для слабых ПК |
| Форматирование: diff исходного и выходного файла | ручная проверка + скрипт diff | Корректность сохранения структуры |
| Субъективная оценка (1–5) | ручная по 10 примерам | Итоговое качество |
